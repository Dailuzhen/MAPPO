## 四、失败原因深度分析

### 4.1 微调未能提升的原因

#### 原因 A：小地图灾难性遗忘

微调地图范围 14-22，8-13 的训练数据为零。即使 lr=1e-5，350 episode 的纯大地图数据足以让小地图策略漂移。

证据：8x8 从 95% 降到 75%，10x10 从 90% 降到 75%。

#### 原因 B：Critic 完全失效（根因）

Explained Variance 全程仅 0.005-0.007（正常应 0.3-0.9）。

PPO 依赖优势函数 A = R - V(s) 来确定梯度方向。当 Critic 预测 V(s) 几乎是随机值时，优势估计退化为纯噪声，PPO 实际变成了 REINFORCE（无基线策略梯度），方差极大，策略更新方向随机。

Critic 失效原因：V8 的 Critic 是在 8-20 地图 + 4 阶段课程分布上学到的，微调突然切到 14-22 均匀分布，旧的价值预测不再适用。而 critic_lr = 1e-5 太低，Critic 来不及适应。

#### 原因 C：obs_radius=3 的容量瓶颈

微调实验证明了一个事实：在 obs_radius=3 架构下，提升大地图性能必然伴随小地图性能下降。模型的"总容量"有限——7x7 视野 + 256 隐藏层的网络无法同时记住所有尺寸的最优策略。

证据：V8 ep1350 在 20x20 有 90%，但 12x12 只有 55%，性能分布极不均匀。微调后变得更均匀（60-80%），但峰值消失。

### 4.2 超时失败的根因分析

V8 ep1350 的失败案例统计：

| 地图 | 失败数 | 超时 | 碰撞 | 超时占比 |
|------|--------|------|------|---------|
| 8x8 | 1 | 1 | 3 | — |
| 10x10 | 2 | 2 | 2 | — |
| 12x12 | 9 | **9** | 0 | **100%** |
| 14x14 | 6 | **6** | 0 | **100%** |
| 16x16 | 7 | **7** | 0 | **100%** |
| 18x18 | 5 | **5** | 0 | **100%** |
| 20x20 | 2 | 2 | 2 | — |
| 22x22 | 10 | 10 | 184 | 特殊 |

**12-18 的所有失败 100% 是超时，零碰撞。** 碰撞避让已经学好，问题出在路径效率。

#### 超时原因 1：Agent 原地不动（stay 动作滥用）

代码中动作空间为 `[stay, up, down, left, right]`，action=0 是 stay。

**环境中存在一个逻辑漏洞**：卡住检测代码只在"部分 agent 到达目标"时才触发：

```python
if any(agents_at_goal) and not all(agents_at_goal):
    # 检测卡住 ...
```

当两个 agent **都没到目标、都选择 stay 原地不动**时，这个检测不会触发。步数照常消耗，直到超时。

Agent 可能在以下情况学会"不动"：
- 面对不确定的局面，stay 不会产生碰撞惩罚（-5）也不会增加距离惩罚
- 相比走错方向的风险，stay 是"最安全"的选择
- 在确定性推理（deterministic=True）下，如果 stay 的概率最高，agent 会永远选 stay

#### 超时原因 2：陷入位置循环

Agent 在几个位置之间反复移动（如 A→B→A→B），位置在变化但没有朝目标前进。当前 anti_stuck 惩罚（-0.1 * visit_count）在训练时有效，但在评估推理时不影响动作选择。

#### 超时原因 3：视野盲区导致的错误决策

obs_radius=3 在 12-18 地图上覆盖率 15-34%，处于"看到一些但看不够全"的区域。agent 可能基于局部视野做出局部最优但全局错误的决策。

- 新增诊断：stay 占比、循环率、超时距离

### 5.4 诊断发现

#### 发现 1：Stay 动作滥用 — 全尺寸系统性问题（已确认）

所有尺寸的 stay 占比 **18%-25%**，即每 4-5 个动作就有一个是原地不动。这不是大地图独有的问题，8x8 的 stay（24%）甚至高于 22x22（18%）。

**根因**：stay 动作（action=0）在训练中是"最安全"的选择——不会触发碰撞惩罚（-5），也不会增加距离惩罚。agent 在不确定时学会了用 stay 来规避风险。

**已修复**：`envs/env.py` 中的卡住检测漏洞已修复，现在所有 agent 连续 3 步不动就会触发任务重新生成（原来只在部分 agent 到达目标时检测）。

#### 发现 2：循环率随地图增大而升高（已确认）

loop 率从 10x10 的 14% 上升到 20x20 的 **26%**。大地图上 agent 更容易在几个位置间反复移动，位置在变但不朝目标前进。

#### 发现 3：超时距离递增 — 大地图上"更迷路"

超时时到目标的平均曼哈顿距离：8x8=3.2 → 22x22=10.6。大地图超时的 agent 离目标更远，说明不仅是"差一步到不了"，而是方向决策出了问题。

#### 发现 4：18x18 碰撞异常

18x18 有 93 次碰撞，远超其他尺寸（1-8 次）。可能是该尺寸下特定场景导致 agent 反复碰撞。

#### 发现 5：成功场景的路径效率极高

所有成功场景的 step ratio 均为 1.00x-1.05x，接近 A\* 最优路径。说明 agent 一旦找到路径，效率极高——**问题不在于路径质量，而在于"能否找到路径"**。

### 5.5 诊断结论

| 问题 | 严重程度 | 数据证据 | 影响范围 |
|------|---------|---------|---------|
| Stay 滥用 | **高** | 全尺寸 18-25% | 所有尺寸 |
| 位置循环 | **中高** | 大地图 23-26%，小地图 14-18% | 主要影响大地图 |
| 超时是主要失败模式 | **高** | 124/400 场景超时 | 所有尺寸 |
| 大地图方向迷失 | **中** | 超时距离 8.8-10.6 | 20x20, 22x22 |
| 18x18 碰撞异常 | **中** | 93 次碰撞 vs 其他 1-8 次 | 仅 18x18 |

## 六、当前模型能力的真实水平（基于 50 样本修正）

### 已确认的能力

- 路径效率极高：成功场景 ratio ≈ 1.0x，接近 A\* 最优
- 纯策略推理：完全不依赖 A\*，策略网络独立完成路径规划
- 碰撞避让基本有效：除 18x18 异常外，碰撞率很低

### 已确认的瓶颈

- 综合成功率 69%（非之前认为的 73.8%），各尺寸 60%-74%
- Stay 动作系统性滥用（18-25%），导致大量超时
- 大地图循环率高（20-26%），agent 在局部区域反复移动
- obs_radius=3 视野不足，大地图上方向决策能力退化

## 七、下一步方案（基于诊断数据更新）

诊断数据已明确：**超时主因是 stay 滥用 + 位置循环**，视野不足是次要因素。

### 方案 D：奖励修复 + V8 重训练（推荐优先执行）

**做法**：
1. 给 stay 动作增加小惩罚（如 -0.05），降低 stay 的"安全优势"
2. 增强 anti_stuck 系数（0.1 → 0.2）或增大历史窗口（20 → 30 步）
3. 利用已修复的卡住检测（连续 3 步不动 → 重新生成任务）
4. 在 V8 架构上从 Phase A ep190 重新跑 Phase B

**依据**：stay 占比 18-25% 是系统性问题，奖励层面未惩罚 stay 是根因。修复后有望将 stay 降到 5% 以下，综合成功率提升到 75%+。

**时间**：6-8 小时（仅 Phase B）。

### 方案 A：单变量 V10 —— 仅提升 obs_radius 到 4

**做法**：保持 V8 所有配置不变，仅将 obs_radius 从 3 改为 4（9x9 视野）。从 Phase A 重新训练。

**依据**：
- 大地图超时距离 8.8-10.6 说明视野不足导致方向决策问题
- obs_radius=4 在 22x22 上覆盖率从 10.1% 提升到 16.7%（+65%）
- 只改一个变量，可以清晰归因效果

**时间**：12-15 小时。

**风险**：输入维度从 55 增加到 87（+58%），可能需要适当增大 hidden_size。

### 方案 B：A\* 行为克隆补丁（快速验证）

**做法**：不用 PPO，在 18-22 大地图上跑 A\* 收集专家演示，用 BC 直接微调 ep1350 模型。

**依据**：BC 不依赖 Critic，绕过 EV=0.005 的问题。Phase A 已证明 BC 在该环境中有效（accuracy 可达 86%）。

**时间**：2-3 小时。

**风险**：BC 天花板有限，可能不如 PPO 的长期效果。

### 推荐执行顺序

```
Step 1: 方案 D — 奖励修复 + Phase B 重训练（6-8 小时）          [已具备条件]
        ↓ 验证修复效果
        ├─ 如果 stay 下降 + 成功率 ≥75% → 继续在此基础上优化
        ├─ 如果成功率提升有限 → 方案 A（obs_radius=4 重训练）
        └─ 如果想快速对比 → 方案 B（BC 补丁，2-3 小时出结果）
```
